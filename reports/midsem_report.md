## ABSTRACT

The AI-Based Recruitment Assistant is a software platform for two-sided
recruitment workflows that simultaneously addresses two well-documented
limitations of conventional applicant tracking systems (ATS): semantic
mismatch in keyword-based candidate retrieval, and the inability to
predict application drop-off in real time. The platform is built as a
modular Python application running on standard CPU hardware, with no
dependency on paid cloud APIs or specialised infrastructure.

The system is composed of three principal machine-learning components
fused into a single recruiter-facing workflow. The Semantic Matching
Module encodes résumés and job descriptions using a pre-trained sentence
transformer (SBERT, `all-MiniLM-L6-v2`), indexes the dense vectors in a
FAISS approximate nearest-neighbour structure, and combines them with a
classical BM25 lexical baseline through reciprocal rank fusion. The
Drop-off Prediction Module is a gradient-boosted XGBoost classifier
trained on 10,000 synthetic candidate-job-session triples generated by a
documented logistic-rule simulation calibrated to the real-world 25%
drop-off prevalence observed in the HR Analytics benchmark, with Optuna
hyperparameter search and isotonic probability calibration. The
Explainer Module produces per-candidate rationales in a structured JSON
format through either a deterministic templated composer or a local
large language model (Llama 3.1 8B via Ollama), both grounded in SHAP
feature attributions and retrieved evidence chunks.

The complete prototype is delivered through two Streamlit interfaces — a
recruiter dashboard for ranked candidate review and a mock
candidate-side application form demonstrating live in-session drop-off
prediction — supported by an end-to-end command-line demonstration
script and a Makefile providing single-command reproducibility.

The system operates on a combined résumé corpus of **2,960 records**
drawn from two sources: the HuggingFace
`cnamuangtoun/resume-job-description-fit` benchmark (477 résumés, the
labelled subset used as ground truth for retrieval metrics) and the
Kaggle `snehaanbhawal/resume-dataset` (2,483 unlabelled résumés across
24 occupational categories used to extend the search pool to realistic
scale). The job-description side draws 1,500 sub-sampled postings from
the Kaggle `arshkon/linkedin-job-postings` corpus (3.4 million records
total) for both retrieval queries and as the JD population for the
drop-off training dataset. A fourth Kaggle dataset, the HR Analytics
job-change corpus (19,158 rows with real `gender` and `target` labels),
supports the non-proxy fairness validation track. The drop-off
classifier is trained over 10,000 synthetic candidate-job-session
triples drawn from this combined résumé × LinkedIn job space.

Quantitative retrieval evaluation against the 477-résumé labelled subset
(61 queries with ≥ 1 relevant item) reports Precision@K, Recall@K,
NDCG@K and MRR for each channel; an extended retrieval ablation on the
full 2,960-résumé pool demonstrates that the channel-ordering
(Dense > BM25; Hybrid ≈ Dense) is preserved at the larger scale.
Drop-off classification achieves ROC-AUC of 0.82, F1 of 0.59, and Brier
score of 0.17 on the held-out test split. A fairness audit conducted on
both synthetic data with documented demographic proxies and real
demographic data from the HR Analytics dataset passes the EEOC 4/5 rule
in both tracks.

An exploratory Experience-Aware Retrieval Channel was additionally
designed, implemented and evaluated as a third candidate-recall channel
fused alongside Dense and Lexical retrieval. The channel scores
candidates by deterministic trajectory features (years of experience,
seniority progression, domain depth, company tier) and is integrated in
two configurations (parallel-recall and post-fusion reranker). An
extractor-quality study reduced false-positive labels in the trajectory
parser by approximately one order of magnitude
(Director-level: 50.5% → 1.7%; Tier-1 employer: 54.7% → 4.0%). With the
improved extractor, the channel still did not surpass the two-channel
baseline on the labelled benchmark — a methodologically interesting
finding that isolates the negative result to the orthogonality of
career-trajectory signal with skill-fit labels rather than to extractor
implementation. This exploratory contribution is reported in the
methodology chapter as a documented negative result with the production
configuration retained as the Hybrid-2 (Dense + BM25) channel.

The remaining work for the dissertation comprises the qualitative
recruiter feedback round with at least three practising recruiters, the
final RAG faithfulness evaluation, and the thesis chapter writing.

**SYSTEM / SUB-SYSTEM DESCRIPTION:** The AI Recruitment Assistant
consists of the following major components:

(a) Semantic Retrieval Module
(b) Drop-off Prediction Module
(c) Explainer Module (templated and LLM-based variants)
(d) Fusion Ranker
(e) Data Pipeline and Embedding Cache
(f) Evaluation Harness
(g) Fairness Audit Module
(h) Recruiter Streamlit Interface
(i) Candidate Live Application Interface
(j) Reproducibility Layer (Makefile)
(k) Experience-Aware Retrieval Channel (exploratory)

The description of each of the major components is explained below:

a) **Semantic Retrieval Module:** Encodes résumés and job descriptions
using a transformer-based sentence-embedding model (Sentence-BERT, with
`all-MiniLM-L6-v2` as the default 384-dimensional encoder). Embeddings
are normalised to unit length and indexed in a FAISS `IndexFlatIP`
structure, where inner-product search on unit vectors is mathematically
equivalent to cosine similarity. In parallel, a BM25 lexical index is
maintained using the `rank-bm25` library, providing the keyword baseline
the dissertation evaluates dense retrieval against. The two retrieval
channels are fused using Reciprocal Rank Fusion with the canonical
smoothing constant *k = 60* from Cormack et al. (2009). Structured hard
filters (location, minimum years of experience, required certifications)
are applied post-fusion.

b) **Drop-off Prediction Module:** An XGBoost gradient-boosted
classifier predicting per-candidate probability of application
abandonment. The model is trained on 10,000 synthetic
candidate-job-session triples generated from a documented logistic-rule
simulation with intercept auto-calibration to a configurable positive
class rate. Twenty-seven features across four families (match,
candidate, job, and session telemetry) are exposed to the model;
demographic features (gender, name, photo, age, race) are explicitly
excluded. Optuna with the Tree-structured Parzen Estimator sampler
performs hyperparameter search over thirty trials; an isotonic
regression on the validation set provides probability calibration.

At inference time the same trained model is used in two distinct
operational modes, corresponding to the two user-facing surfaces.

- **Pre-application ranking mode (recruiter view, module h).** When a
  recruiter pastes a job description and the system ranks candidates
  before any application activity exists, real session telemetry is
  unavailable by definition. The eight session features
  (`session_time_on_page_secs`, `session_n_fields_skipped`, navigation
  back-count, mobile/desktop, hour-of-day, etc.) are **simulated** for
  each candidate from the same distributions used at training time. The
  resulting prediction is interpreted as *"if this candidate were to
  attempt the application with a typical session pattern, what is the
  expected drop-off risk?"* — a hypothetical-session estimate used to
  rank candidates, not a measurement of a real in-flight session.
- **In-session live mode (candidate view, module i).** When a candidate
  is actively filling the mock application form, real telemetry from
  field interaction events is streamed into the same model after each
  interaction. The prediction is then a genuine in-session early-warning
  signal of the dissertation's stated production use case.

The dissertation reports these two modes explicitly. A real production
deployment would require integration with an Applicant Tracking System
exposing live form-event telemetry; this is identified as future work.

c) **Explainer Module:** Generates per-candidate natural-language
rationales in a structured JSON schema. Two implementations of the same
schema are provided: a deterministic templated composer producing
rationales from skill set intersections, fit-band rules, and SHAP
attribution text; and a Retrieval-Augmented Generation variant calling a
local large language model (`llama3.1:8b-instruct` served by Ollama) with
the same evidence sources, constrained to JSON output and configured to
fall back to the templated composer on any error. SHAP TreeExplainer
attributions on the trained XGBoost model are textualised and inserted
into the LLM prompt as structured context — referred to as the SHAP→RAG
bridge — to prevent ungrounded LLM hallucination.

d) **Fusion Ranker:** Combines the semantic-similarity score, drop-off
probability, and optional cross-encoder reranking score into a single
final-candidate score per the weighted formula
*final = w_sem · semantic + w_stay · (1 − P_dropoff)*, with weights
exposed as adjustable parameters in the Streamlit recruiter interface.
When used for pre-application ranking, the drop-off term *P_dropoff* is
the simulated-session prediction described in module (b) — a
hypothetical-session estimate rather than a real-time measurement. The
Fusion Ranker is the production ranking surface for the recruiter view;
the Candidate Live UI (module i) consumes the in-session live drop-off
prediction directly without the fusion step.

e) **Data Pipeline and Embedding Cache:** Pydantic schemas (`Resume`,
`Job`, `Match`, `DropoffSample`) enforce typed contracts across module
boundaries. Loaders are implemented for sample CSV data, the HuggingFace
`cnamuangtoun/resume-job-description-fit` labelled benchmark, the Kaggle
`snehaanbhawal/resume-dataset` (2,483 résumés across 24 occupational
categories), the Kaggle LinkedIn job postings dataset (3.4 million
records with deterministic sub-sampling), and the Kaggle HR Analytics
dataset (19,158 rows with real demographic labels). SBERT embeddings are
cached on disk by content hash to make repeated runs instantaneous.

f) **Evaluation Harness:** Implements pure-function information retrieval
metrics — Precision@K, Recall@K, NDCG@K (with exponential gain), Mean
Reciprocal Rank — operating over graded relevance labels (0 = No Fit,
1 = Potential Fit, 2 = Good Fit). An orchestrator runs any retriever
matching the `Callable[[str, int], list[int]]` interface against a query
set and reports per-query and aggregate metrics.

g) **Fairness Audit Module:** Two parallel audit tracks. The retrieval
audit computes Skew@K and Normalised Discounted KL divergence across
demographic groups for each retrieval channel. The classifier audit
computes demographic-parity difference, equal-opportunity difference and
disparate-impact ratio using the `fairlearn` library, evaluating against
both name-based gender / location-based country proxies (synthetic data
track) and real `gender` labels from the HR Analytics dataset
(real-demographic track).

h) **Recruiter Streamlit Interface:** A web-based recruiter dashboard
served on `http://localhost:8501`. The interface accepts a pasted or
selected job description, displays the top-K ranked candidates as cards
showing semantic score, calibrated drop-off probability, fused final
score, matched and missing skills, SHAP-grounded rationale, an
expandable SHAP waterfall figure, and an action recommendation
(Interview / Consider / Likely Mismatch). A model-comparison toggle
displays two trained variants side-by-side per candidate.

i) **Candidate Live Application Interface:** A second Streamlit
application served on `http://localhost:8502` simulating the
candidate-side application form. As the candidate interacts with the
form, real session telemetry (time-on-page, fields completed and
skipped, navigation-back count, device class) is fed live to the
Drop-off Prediction Module, demonstrating the production use case of
in-session early-warning detection.

j) **Reproducibility Layer:** A `Makefile` exposes nineteen file-existence
guarded targets covering installation, dataset download, model training
and tuning, evaluation, fairness audit, and demo launch. The composite
`make reproduce` target re-runs the full pipeline end-to-end from a
clean checkout in approximately five minutes on warm cache.

k) **Experience-Aware Retrieval Channel (exploratory):** A third
candidate-recall channel implemented alongside Dense and Lexical
retrieval and reported as an exploratory contribution. The module
extracts structured trajectory features per résumé — years of experience
(YOE), seniority level (Junior/Mid/Senior/Staff/Director),
primary domain (backend/frontend/data/ml/devops/mobile/security),
tenure progression, and a company-tier indicator — and a parallel
JobCriteria schema is extracted from each JD. A composite trajectory
score combines YOE-match, seniority-match, domain overlap, tenure depth,
and an optional, fairness-auditable company-tier bonus. The channel is
evaluated in two integration configurations: parallel-recall via N-channel
Reciprocal Rank Fusion (with and without channel weighting), and a
trajectory-aware reranker applied to the post-fusion candidate pool.

The trajectory extractor was developed in two generations. The initial
rules-based extractor exhibited systematic false positives on the HF
corpus — approximately 50% of résumés classified as Director-level and
55% flagged as having a Tier-1 employer, attributable to substring
matching for seniority keywords and bare-word matching for company
names. A diagnostic-driven second generation reduced these
false-positive rates to 1.7% and 4.0% respectively by requiring
employment-context phrasing for company-name matches and engineering-role
adjacency for seniority words, and by expanding the YOE-detection
pattern set from three to ten regex patterns. With the improved
extractor, the channel produced a marginal Precision@5 improvement
(0.125 vs the Hybrid-2 baseline of 0.121) but did not surpass the
baseline at K=10 on Precision, Recall or NDCG. This isolates the
negative result to the orthogonality of trajectory signal with the
skill-fit labels in the benchmark dataset, rather than to extractor
implementation quality.

The channel is retained in the dissertation codebase and methodology
chapter as a documented exploratory contribution. The production
retrieval configuration shipped in the recruiter interface remains the
Hybrid-2 (Dense + BM25) channel. Future work directions identified
include trajectory-labelled evaluation data (the present benchmark labels
skill-fit and is orthogonal to trajectory) and LLM-assisted structured
extraction to address the remaining 66% of résumés with no detectable
YOE phrasing under the rules-based parser.

## 2. FUNCTIONAL BLOCK DIAGRAM / DESCRIPTION OF THE AI RECRUITMENT ASSISTANT

The data and control flow during a recruiter session is shown in
Figure 2. A pasted job description enters the system through the
Streamlit interface and is processed by the Semantic Retrieval Module,
which encodes it to a 384-dimensional vector via SBERT and concurrently
tokenises it for BM25. Dense and lexical top-K hits are produced
independently, fused via Reciprocal Rank Fusion, and filtered against
structured hard requirements. For each remaining candidate, the Drop-off
Prediction Module assembles a 27-feature row from the retrieval signals,
résumé-derived trajectory features and a simulated or live session
telemetry block, computes a calibrated drop-off probability through the
trained XGBoost pipeline, and produces a SHAP attribution vector. The
Explainer Module composes a per-candidate JSON rationale grounded in
matched and missing skills, the SHAP drivers and (optionally) the LLM's
language model. The Fusion Ranker combines the semantic and stay-
probability components into a single score, and the top-K results are
rendered as cards in the Streamlit recruiter view.

**Figure 2: Functional block diagram of the end-to-end pipeline**

The dashed Experience-Aware Retrieval Channel branch on the retrieval row
indicates the exploratory third channel that was evaluated alongside the
production Dense + BM25 fusion. The shipped retrieval configuration uses
only the solid-line path; the dashed branch is retained as a documented
exploratory contribution.

**KEY DATA FLOWS:**

1. **Embedding Cache:** SBERT vectors are encoded once per corpus and
   persisted to disk indexed by a content-derived hash. Subsequent runs
   load from cache, eliminating repeated GPU/CPU encoding cost.

2. **Model Loading:** The drop-off classifier is loaded into memory at
   Streamlit startup via `@st.cache_resource`; six trained model
   variants (v0/v1 × default/tuned/calibrated) are exposed through a
   sidebar selector to allow examiner-time comparison.

3. **Live Telemetry Stream:** In the candidate-side application,
   field-level interaction events update Streamlit session state on
   every rerun; aggregated session features are recomputed and the
   drop-off prediction re-evaluated on each interaction, producing a
   visibly live risk badge with sub-second response.

4. **Inference Path:** End-to-end inference for one job description on
   the 2,960-résumé pool with top-K = 5 completes in approximately three
   seconds on warm cache (Apple M1, no GPU), inclusive of retrieval,
   thirty drop-off predictions, SHAP attribution and explanation
   composition.

5. **Fallback Behaviour:** The LLM RAG explainer falls back
   deterministically to the templated composer if the Ollama service is
   unreachable or returns malformed JSON, guaranteeing the system
   degrades gracefully during demonstrations.

## 4. DESIGN CONSIDERATIONS

- **Public and synthetic data only.** No personally identifiable
  information from real candidates is used at any stage. All datasets
  are publicly available under permissive licences (CC0-1.0 in the case
  of the Kaggle corpora).
- **Drop-off labels are synthetic.** Real-world ATS data with
  application-level drop-off labels is not publicly available. The
  dissertation handles this transparently: the simulation rule is
  documented in `docs/concepts/04_dropoff_design.md`, the rule
  intercept is auto-calibrated to match the 25% prevalence observed in
  the HR Analytics dataset, and a parallel real-label classifier is
  trained on the HR Analytics target for validation comparison.
- **Drop-off prediction operates in two modes.** The same trained
  classifier serves two functionally distinct purposes: a
  *pre-application ranking* mode used by the recruiter view (which
  ranks candidates before they have applied, using simulated session
  features drawn from the training distribution), and an *in-session
  live* mode used by the candidate view (where real form-interaction
  telemetry is streamed into the same model). The distinction is made
  explicit in module (b); the Fusion Ranker uses only the
  pre-application mode and the Candidate Live UI uses only the
  in-session mode. Real-world production deployment of the in-session
  mode would require integration with an Applicant Tracking System
  exposing live form events — identified as future work.
- **Demographic features are excluded from the model.** Gender, name,
  photo, age and race are explicitly omitted from the drop-off feature
  set. The fairness audit measures indirect bias arising through
  correlated features rather than direct discrimination.
- **Reproducibility is a first-class deliverable.** A `Makefile`
  exposes all artefact-producing targets with file-existence guards;
  `make reproduce` reruns the entire pipeline end-to-end from a clean
  checkout. All random seeds are fixed at `seed = 42` throughout.
- **Local LLM, no API costs.** The Retrieval-Augmented Generation
  explainer uses a locally hosted Llama 3.1 8B model via Ollama. The
  dissertation has no dependency on paid LLM APIs (OpenAI, Anthropic) at
  any stage.
- **Honest negative results retained.** An exploratory
  Experience-Aware Retrieval Channel was built, evaluated and shown to
  not improve retrieval metrics on the labelled benchmark even after a
  significant extractor-quality improvement. This negative result is
  reported as an exploratory contribution in the methodology chapter
  rather than removed; the documented finding is that career-trajectory
  signal is orthogonal to skill-fit labels on this particular benchmark.
- **Single-threaded numerical libraries.** The launch command sets
  `OMP_NUM_THREADS=1` to avoid intermittent macOS segmentation faults
  arising from concurrent OpenMP allocation between FAISS, XGBoost and
  scikit-learn.
